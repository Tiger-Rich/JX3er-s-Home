# 番薯万事屋 MVP 设计规格

## 1. 项目定位

番薯万事屋是面向《剑侠情缘网络版3》老玩家的身份信任型资源对接产品。它不是游戏攻略社区，也不是交易担保平台，而是帮助散落在各行各业的番薯以较低成本找到彼此，并完成求职、内推、行业咨询、轻量委托、本地互助等资源对接。

第一阶段交付一个可运行的轻量全栈 Web 原型，移动端优先，体验上接近微信小程序。核心主线聚焦求职、内推、行业咨询；其他委托类型保留入口，但不在首版做复杂交易、订单、担保或评价体系。

## 2. 核心原则

- 所有用户都必须有可审核的番薯身份，不允许匿名发布。
- 注册认证必须提供游戏区服和游戏 ID/昵称。
- 产品不接触游戏账号密码，不要求用户提供敏感账号凭据。
- 联系方式默认隐藏，只有联系申请通过后双方互相可见。
- 信息流只展示审核通过的委托。
- 首版强调人工审核和明确边界，不承诺求职成功、交易成功或担保结果。
- 文案贴合《剑网3》的江湖、门派、亲友、侠义语境，但保持现代、清爽、可信，不使用官方素材、剧情台词、NPC 形象或可能侵犯权益的专有表达。

## 3. 首版目标用户

- 《剑网3》老玩家，通常有较强身份认同和稳定社群关系。
- 30 岁上下或以上，已有职业身份、行业经验或现实资源。
- 愿意基于同为番薯的信任感进行轻量互助。
- 主要需求包括求职内推、行业咨询、职业资源对接；次要需求包括农产品/特产买卖、约稿委托、本地搭子、追星互助、二手交易、服务推荐等。

## 4. 产品结构

用户端包含四个底部导航：

1. 万事广场
2. 发个委托
3. 联系申请
4. 我的名片

后台使用独立的 /admin 管理界面，命名保持专业，不做江湖化包装。

核心闭环：

1. 用户注册并填写我的名片。
2. 用户提交番薯身份认证。
3. 管理员审核认证。
4. 已认证用户发布委托。
5. 管理员审核委托。
6. 委托进入万事广场。
7. 已认证用户递交联系申请。
8. 发布者确认后，双方可见联系方式。

## 5. 用户端设计

### 5.1 万事广场

万事广场展示已审核通过的委托信息流，默认优先展示求职内推和行业咨询。

筛选项：委托类型、城市、行业、是否远程。

委托卡片展示：标题、委托类型、城市/远程、行业、有效期、发布者昵称、区服、门派、认证状态。

页面气质短句：同在江湖，先看身份，再谈合作。

### 5.2 委托详情

委托详情展示完整需求和发布者公开身份。

需求信息：标题、类型、描述、城市/远程、预算或回报、有效期。

发布者公开身份：昵称、区服、门派、入坑年份、城市、行业、已确认番薯身份。

操作：递交联系申请、收藏、举报。

联系方式不会在详情页直接展示。申请说明占位文案：简单说说你是谁、能帮什么，或想聊什么。

### 5.3 发个委托

已认证用户可以发布委托。未认证、待认证、认证未通过用户不能发布。

类型：求职内推、行业咨询、买卖交易、约稿委托、本地互助、追星互助、其他。

必填字段：类型、标题、描述、城市/是否远程、有效期。

选填字段：预算或回报。

产品规则：不提供匿名开关；发布后进入待审核；审核通过后进入万事广场；审核拒绝时显示后台填写的拒绝原因。

边界提示：万事屋不接账号交易、代练、外挂、私服相关委托，也不承诺求职或交易结果。

### 5.4 我的名片

我的名片用于展示和审核用户身份与现实资源。

游戏身份字段：昵称、区服、游戏 ID/昵称、门派、入坑年份。

现实资源字段：城市、职业/行业、我能提供什么、我正在寻找什么。

联系方式：微信、手机号或邮箱等至少一种联系方式；默认隐藏；仅在联系申请通过后对双方可见。

认证状态：未提交、待认证、已确认番薯身份、未通过。

认证说明：请填写区服与游戏ID/昵称，用于确认你确是同路番薯。我们不会索要账号密码。

辅助材料：角色截图说明、入坑问题回答、亲友邀请说明、其他人工审核备注。

### 5.5 联系申请

联系申请分为我递出的申请和我收到的申请。

我递出的申请状态：待确认、已通过、被拒绝。

我收到的申请：查看申请人的公开名片和一句话说明；通过后双方展示联系方式；拒绝后不展示联系方式。

按钮文案：通过为同意见面聊聊；拒绝为暂不合适。

空状态文案：暂时还没有新的联系申请。江湖路远，慢慢来。

## 6. 管理后台设计

后台用于审核、风控和基础运营。

### 6.1 认证审核

管理员查看用户提交的区服、游戏 ID/昵称、门派、入坑年份、辅助材料说明和联系方式。操作为通过或拒绝。拒绝必须填写原因，原因会展示给用户。

### 6.2 委托审核

管理员查看待审核委托。操作为通过、拒绝、下架。拒绝和下架必须填写原因。

### 6.3 用户列表

支持按昵称、区服、城市、行业、认证状态、用户状态搜索或筛选。管理员可以禁用用户。禁用后用户不能使用核心功能，其已发布委托从信息流隐藏。

### 6.4 委托列表

支持按类型、城市、行业、审核状态、是否过期筛选。管理员可以下架违规委托。

### 6.5 联系申请与举报

后台可以查看联系申请状态，用于处理举报或纠纷，但默认不替用户代操作。

举报首版并入委托和用户管理流程：用户端提供举报入口；后台记录举报对象、举报原因、处理状态；管理员可基于举报下架委托或禁用用户。

## 7. 权限规则

未登录用户：可浏览万事广场和委托详情；不能发布委托、申请联系、收藏。

已登录未认证用户：可填写我的名片、提交认证、浏览内容；不能发布委托、申请联系、收藏。

待认证用户：可继续完善资料；不能发布委托、申请联系。

已认证用户：可发布委托、申请联系、收藏委托、处理收到的联系申请。

管理员：可审核认证、审核委托、下架内容、处理举报、禁用用户。

被禁用用户：不能使用核心功能；历史委托从信息流隐藏。

## 8. 数据模型

### User

字段：id、account、passwordHash、nickname、city、contactValue、contactVisibleAfterApproval、role、status、openid、createdAt、updatedAt。

role 取值为 user 或 admin。status 取值为 active 或 disabled。openid 预留给后续微信小程序迁移。

### Profile

字段：id、userId、server、gameNickname、sect、startedYear、industry、occupation、canOffer、lookingFor、createdAt、updatedAt。

### Verification

字段：id、userId、status、supportMaterial、reviewerId、reviewedAt、rejectReason、createdAt、updatedAt。

status 取值为 not_submitted、pending、approved、rejected。

### Request

字段：id、ownerId、type、title、description、city、remote、industry、budgetOrReward、expiresAt、status、rejectReason、takedownReason、createdAt、updatedAt。

status 取值为 draft、pending、approved、rejected、taken_down、expired。

### ContactApplication

字段：id、requestId、applicantId、ownerId、message、status、handledAt、createdAt、updatedAt。

status 取值为 pending、approved、rejected。

### Favorite

字段：id、userId、requestId、createdAt。

### Report

字段：id、reporterId、targetType、targetId、reason、status、handlerId、handledAt、resultNote、createdAt。

targetType 取值为 request、user、contact_application。status 取值为 pending、resolved、dismissed。

## 9. 状态流转

认证状态：未认证 -> 待认证 -> 已认证；待认证也可以进入未通过；未通过修改后可重新提交；已认证用户可被管理员禁用。

委托状态：草稿 -> 待审核 -> 已发布；待审核也可以进入已拒绝；已发布可以被下架或因超过有效期进入已过期。

联系申请状态：待确认 -> 已通过或已拒绝。

## 10. 技术架构

首版使用轻量全栈 Web 原型：

- 前端：React + Vite。
- UI：移动端优先，模拟微信小程序底部导航和页面层级；后台适配桌面。
- 后端：Node.js + Express。
- 数据库：SQLite。
- 鉴权：简单账号登录，保留 openid 字段用于后续微信小程序迁移。
- 路由：用户端和 /admin 后台在同一项目中。

后续迁移到微信小程序时，优先复用数据模型、接口语义、审核状态流转、权限规则和页面信息架构。

## 11. 首版实现范围

包含：用户注册/登录、我的名片填写与编辑、番薯身份认证提交、后台认证审核、委托发布、后台委托审核、万事广场信息流、委托详情、收藏、递交联系申请、发布者处理联系申请、通过后双方查看联系方式、举报入口、后台下架和禁用用户。

暂缓：微信原生小程序工程、微信登录、订阅消息、云开发、支付、担保交易、订单系统、即时聊天、复杂信誉分和评价体系、自动 OCR 或截图识别认证、广告位和付费推广。

## 12. 风险与边界

禁止内容：账号交易、代练、外挂、私服、虚假招聘、诈骗引流、违法交易、明示或暗示平台担保求职、交易或服务结果。

首版通过人工认证审核、委托发布审核、联系方式申请制、举报入口、管理员下架、管理员禁用用户和清晰边界提示控制风险。

## 13. 验收标准

核心流程可完整跑通：注册/登录 -> 填写我的名片 -> 提交认证 -> 后台通过认证 -> 发布委托 -> 后台通过委托 -> 委托展示在万事广场 -> 另一名已认证用户递交联系申请 -> 发布者通过申请 -> 双方可见联系方式。

权限验证：未登录不能发布、申请、收藏；未认证不能发布、申请；待认证不能绕过限制；联系方式不会在申请通过前展示；被禁用用户不能使用核心功能。

审核验证：认证拒绝原因可见；委托拒绝原因可见；下架后委托从信息流隐藏；过期委托不再作为有效委托展示。

体验验证：360px 左右移动端宽度下页面可正常使用；表单、按钮、筛选项不拥挤；后台列表筛选和审核操作清楚可用；文案有轻江湖感，但不影响严肃资源对接的可信度。

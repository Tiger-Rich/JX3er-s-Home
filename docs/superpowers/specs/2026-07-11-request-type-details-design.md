# 委托类型化字段与买卖图片设计

## 1. 背景

番薯万事屋当前的发委托表单采用统一字段：委托类型、标题、委托说明、城市、是否远程、行业、预算或回报、有效期。这个结构适合 MVP 起步，但不同类型的委托需要表达的信息差异很大。求职内推需要岗位和求助方向，买卖交易需要物品、价格、交易方式和图片，本地互助需要时间地点，约稿委托需要交付物和周期。

本次设计目标是把“发个委托”从通用文本发布，升级为按类型填写的资源对接表单，同时保持 MVP 的轻量可信、不做交易担保、不承诺结果的边界。

## 2. 设计原则

- 委托类型从 7 类调整为 6 类，删除“追星互助”。
- 发布表单不再让用户直接填写必填的通用“委托说明”。
- 所有委托类型保留选填的“补充说明”，用于人情语境、背景、特殊注意事项。
- 后端保留 `description` 字段，用系统摘要兼容信息流、详情页、后台审核和旧数据。
- 新增 `details` JSON 字段，保存不同委托类型的结构化字段。
- 买卖交易本轮支持图片上传、预览、删除、详情展示和后台审核展示。
- 图片上传只开放给“买卖交易”，其他类型本轮不开放图片。
- 委托详情页显示反诈提醒，买卖交易显示更强风险提醒。
- 不做账号交易、代练、外挂、私服相关内容，不做支付、担保、订单、评价和售后仲裁。

## 3. 委托类型

保留类型：

- 求职内推：`job_referral`
- 行业咨询：`industry_consulting`
- 买卖交易：`trade`
- 约稿委托：`commission`
- 本地互助：`local_help`
- 其他：`other`

删除类型：

- 追星互助：`fandom_help`

删除后的兼容规则：

- 新发布时不再出现“追星互助”选项。
- 后端不再接受新的 `fandom_help` 委托。
- 如果本地旧数据中已有 `fandom_help`，可在迁移时改为 `other`，后台和详情页按“其他”展示。

## 4. 通用字段

所有类型共有以下字段：

| 字段 | 存储 | 必填 | 用途 |
| --- | --- | --- | --- |
| 委托类型 | `type` | 是 | 决定动态字段、筛选、展示标签 |
| 标题 | `title` | 是 | 信息流和详情页主标题 |
| 城市 | `city` | 条件必填 | 非远程委托必须填写 |
| 可远程 | `remote` | 否 | 勾选后城市可选 |
| 有效期 | `expiresAt` | 是 | 过期后不在信息流展示 |
| 补充说明 | `details.extraNote` | 否 | 用户补充背景、边界、注意事项 |

不再在前端表单中展示原“委托说明”作为必填大文本。`description` 由系统根据 `type`、`details`、`extraNote` 自动生成摘要。

## 5. 类型化字段

### 5.1 求职内推

必填字段：

- 目标岗位：`targetRole`
- 目标行业：`targetIndustry`
- 当前阶段：`careerStage`
- 希望获得的帮助：`helpWanted`

选填字段：

- 期望公司/方向：`targetCompany`
- 简历亮点：`resumeHighlights`

字段说明：

- `careerStage` 用普通文本，不做复杂枚举，允许填写“应届/3年产品/转行中/想看机会”等。
- `helpWanted` 鼓励填写“内推、简历建议、面试经验、岗位信息、行业判断”等。
- 信息流行业筛选使用 `targetIndustry` 同步到通用 `industry` 字段，便于延续现有筛选能力。

### 5.2 行业咨询

必填字段：

- 咨询方向：`topic`
- 具体问题：`questions`
- 期望交流方式：`preferredFormat`

选填字段：

- 我的背景：`background`
- 希望对方资历：`expectedPeer`
- 可提供回报：`reward`

字段说明：

- `questions` 可以是多问题文本，不拆成复杂问卷。
- `preferredFormat` 用普通文本，允许填写“微信文字、语音、腾讯会议、线下咖啡”等。
- 信息流行业筛选使用 `topic` 同步到通用 `industry` 字段，保持首版筛选逻辑简单。

### 5.3 买卖交易

必填字段：

- 物品/服务名称：`itemName`
- 价格或交换方式：`price`
- 成色/规格：`condition`
- 交易/发货方式：`deliveryMethod`

选填字段：

- 是否可议价：`negotiable`
- 售后边界：`afterSalesBoundary`
- 补充说明：`extraNote`

图片字段：

- 图片列表：`images`
- 每个委托最多 6 张图片。
- 单张图片最大 5MB。
- 支持 JPG、PNG、WebP。
- 图片只绑定买卖交易委托，不开放给其他类型。

图片交互：

- 发布页选择图片后立即显示本地预览。
- 用户可以删除已选图片。
- 图片与委托一起提交，委托未发布成功时不产生可见图片。
- 详情页在标题、类型和核心字段之后展示图片。
- 后台审核页显示图片缩略图，方便掌柜判断是否违规。

风控边界：

- 禁止账号交易、代练、外挂、私服相关交易。
- 禁止要求平台担保、先款担保、官方背书等表述。
- 涉及定金、代付、私下链接、异常低价时详情页加强提醒。

### 5.4 约稿委托

必填字段：

- 委托内容：`commissionContent`
- 交付物：`deliverables`
- 预算：`budget`
- 期望交付时间：`deadline`

选填字段：

- 风格参考：`styleReference`
- 使用场景：`usage`
- 商用/非商用：`commercialUse`

字段说明：

- 本轮不上传参考图，只使用文本描述或外部描述。
- 不做合同、付款、验收、排期管理。

### 5.5 本地互助

必填字段：

- 互助事项：`helpTask`
- 地点/区域：`area`
- 时间窗口：`timeWindow`
- 需要几人：`headcount`

选填字段：

- 费用 AA/回报：`costShare`
- 安全注意事项：`safetyNote`

字段说明：

- 本地互助强调现实见面风险，详情页显示通用反诈和安全提醒。
- 不做线下活动报名系统，不收集身份证、精确住址等敏感信息。

### 5.6 其他

必填字段：

- 事情类型：`requestKind`
- 希望对方怎么帮：`helpWanted`
- 回报方式：`reward`

选填字段：

- 背景说明：`background`
- 限制条件：`constraints`

字段说明：

- “其他”用于无法归类但仍符合平台边界的轻量互助。
- 后台审核重点检查是否绕过禁区发布账号交易、代练、外挂、私服、担保交易等内容。

## 6. 数据模型

`requests` 表新增：

- `details TEXT NOT NULL DEFAULT '{}'`：保存 JSON 字符串。

买卖交易图片新增表：

- `request_images`

字段：

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `requestId INTEGER NOT NULL`
- `url TEXT NOT NULL`
- `mimeType TEXT NOT NULL`
- `sizeBytes INTEGER NOT NULL`
- `sortOrder INTEGER NOT NULL`
- `createdAt TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP`

图片存储策略：

- 原型阶段图片保存到本地项目内的上传目录，由 Express 静态路由提供访问。
- 文件名使用服务端生成的不可预测名称，不使用用户原始文件名作为存储文件名。
- 数据库保存可访问 URL、MIME 类型、大小和排序。
- 删除草稿或发布失败时不暴露图片。

`description` 生成规则：

- 服务端根据 `details` 生成一段 1-3 句摘要。
- 摘要用于信息流、详情页简介和后台审核的快速扫读。
- `extraNote` 如果存在，摘要末尾追加“补充说明：……”。
- 旧接口仍返回 `description`，保证现有列表和详情页不会断裂。

## 7. 页面设计

### 7.1 发个委托

流程：

1. 用户选择委托类型。
2. 表单根据类型切换结构化字段。
3. 通用字段始终显示：标题、城市、可远程、有效期、补充说明。
4. 买卖交易额外显示图片上传组件。
5. 提交后进入待审核。

文案：

- 标题仍为“发个委托”。
- 原“委托说明”改为“补充说明”，并标明“选填”。
- 未认证用户提示保持红字：“请点击我的名片提交认证”。
- 边界提醒保留：“万事屋不接账号交易、代练、外挂、私服相关委托，也不承诺求职或交易结果。”

错误处理：

- 缺少必填结构化字段时，在页面反馈明确字段名称。
- 非远程且未填城市时提示填写城市或选择可远程。
- 买卖交易图片超过 6 张时阻止提交。
- 图片格式不是 JPG、PNG、WebP 时阻止提交。
- 单张图片超过 5MB 时阻止提交。

### 7.2 万事广场

信息流卡片展示：

- 类型
- 标题
- 城市/远程
- 行业，仅求职内推和行业咨询展示
- 有效期
- 发布者公开身份
- 系统摘要 `description`

筛选：

- 类型筛选移除“追星互助”。
- 行业筛选只对求职内推和行业咨询自然生效。
- 城市、远程筛选保留。

### 7.3 委托详情

详情页展示顺序：

1. 返回按钮
2. 类型标签
3. 标题
4. 系统摘要
5. 类型化字段详情
6. 买卖交易图片，只有买卖交易且存在图片时展示
7. 有效期、城市/远程、预算或回报等关键元信息
8. 发布者名片
9. 反诈和安全提醒
10. 联系申请、收藏、举报

通用提醒：

“请谨慎甄别委托信息，勿提前转账，谨防上当受骗。平台不提供交易担保。”

买卖交易额外提醒：

“涉及定金、代付、私下链接、异常低价时请提高警惕。万事屋不提供交易担保或售后仲裁。”

提醒位置：

- 放在发布者名片之后、联系申请区域之前。
- 这样用户在决定联系前一定会看到，但不打断阅读主要委托信息。

### 7.4 后台委托审核

后台列表展示：

- 类型、标题、系统摘要
- 类型化字段摘要
- 买卖交易图片缩略图
- 城市/远程、有效期
- 发布者公开身份
- 审核操作

审核要求：

- 后台能看到 `details` 中的结构化字段。
- 买卖交易能看到所有上传图片缩略图。
- 审核拒绝和下架仍必须填写理由。
- 审核重点提示禁区内容：账号交易、代练、外挂、私服、诈骗引流、担保暗示。

## 8. API 设计

发布委托接口：

- `POST /api/requests`
- 请求体新增 `details`
- 买卖交易图片随发布请求使用 `multipart/form-data` 一次提交，服务端接收字段、生成摘要、保存图片并创建委托。

接口校验：

- `type` 必须是 6 个允许类型之一。
- `details` 必须是对象。
- 每种类型校验对应必填字段。
- `description` 由服务端生成或校验为系统生成值，不信任客户端自由文本。
- `fandom_help` 返回 400。
- 非买卖交易提交图片返回 400。
- 买卖交易图片超过限制返回 400。

详情接口：

- `GET /api/requests/:id`
- 返回 `details` 和 `images`

列表接口：

- `GET /api/requests`
- 返回 `description`、必要的 `details` 摘要字段和买卖交易封面图。

后台接口：

- `GET /api/admin/requests`
- 返回完整 `details` 和图片缩略图 URL。

## 9. 测试范围

单元和接口测试：

- 请求类型常量不包含 `fandom_help`。
- 后端拒绝 `fandom_help`。
- 后端接受每种类型的合法 `details`。
- 后端拒绝每种类型缺少必填字段的 `details`。
- `description` 可由 `details` 生成。
- 买卖交易可绑定 1-6 张合法图片。
- 非买卖交易不能绑定图片。
- 非法图片格式、超大图片、超数量图片被拒绝。
- 详情接口返回 `details` 和 `images`。
- 后台接口返回 `details` 和图片信息。

前端测试：

- 发布页按类型切换字段。
- 原“委托说明”不再作为必填字段出现。
- 所有类型显示选填“补充说明”。
- 删除“追星互助”选项。
- 买卖交易显示图片上传组件。
- 图片预览和删除可用。
- 委托详情显示类型化字段。
- 委托详情显示通用反诈提醒。
- 买卖交易详情显示额外反诈提醒和图片。
- 后台审核页显示买卖交易图片缩略图。

端到端测试：

- 已认证用户发布求职内推，管理员审核后出现在万事广场。
- 已认证用户发布买卖交易并上传图片，管理员审核后详情页显示图片和反诈提醒。
- 未认证用户不能发布委托。

## 10. 验收标准

- 发委托页面不再出现必填的通用“委托说明”。
- 发委托页面根据 6 类委托展示不同结构化字段。
- “追星互助”不再出现在发布、筛选、后台审核筛选中。
- 所有类型都有选填“补充说明”。
- 买卖交易支持图片上传、预览、删除、详情展示和后台缩略图展示。
- 买卖交易图片限制为最多 6 张、单张 5MB、JPG/PNG/WebP。
- 委托详情页在联系申请区域前显示反诈提醒。
- 买卖交易详情页显示额外交易风险提醒。
- 后台能审核类型化字段和买卖交易图片。
- 现有注册、认证、发布审核、联系申请、收藏、举报流程保持可用。
- 所有自动化测试、构建和 e2e 验证通过。

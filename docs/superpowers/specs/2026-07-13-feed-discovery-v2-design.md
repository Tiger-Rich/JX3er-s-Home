# 万事广场 V2 发现与推荐设计

## 1. 背景

万事广场当前以单列列表展示委托，适合早期内容量较少时使用。随着委托数量增加，用户会遇到两个问题：

- 很难快速刷到自己想看的委托。
- 不同类型委托的关键信息不同，统一列表卡片无法让用户一眼判断是否值得点进详情。

本次设计目标是把万事广场升级为“找得到优先”的发现页：用户先能快速定位目标委托，再逐步增强可刷性和停留感。

“我的委托 / 我发布的委托管理”暂缓开发，作为后续 backlog 保留，并在万事广场 V2 完成后重新提醒讨论。

## 2. 产品原则

- 万事广场优先解决“更快找到目标委托”，其次考虑“更愿意刷”。
- 展示形态可以借鉴小红书的轻卡片流，但信息结构必须服务资源对接，不做纯内容种草流。
- 卡片负责帮助用户判断是否值得点进详情，详情页负责完整说明。
- 每类委托使用不同卡片摘要模板，不再所有类型共用一套展示字段。
- 卡片信息要直观、清晰，最多展示 3 个类型关键信息点。
- 心形反馈只使用图标和数字，不出现“点赞”二字。
- 推荐排序先做简单、透明、可解释的 MVP 版本，不做复杂个性化算法。
- 推荐不能让热门委托长期霸榜，也不能让买卖交易因图片更吸引点击而压过求职、咨询等严肃需求。

## 3. 范围

本轮包含：

- 万事广场改为双列卡片流。
- 顶部增加频道切换。
- 每张委托卡片支持心形反馈和数量展示。
- 新增推荐排序，保留最新排序。
- 按类型展示不同卡片摘要。
- API 返回心形数量和当前用户是否已点心形；推荐分仅用于服务端排序，不返回给前端展示。

本轮不包含：

- 完全个性化推荐。
- 评论、转发、关注、私信。
- 心形用户列表。
- 复杂反作弊系统。
- “我的委托 / 我发布的委托管理”。
- 付费推广或广告排序。

## 4. 页面结构

万事广场 V2 页面从上到下：

1. 页面标题和短文案。
2. 频道栏。
3. 轻筛选入口。
4. 排序切换。
5. 双列卡片流。
6. 空状态、加载状态、错误状态。

频道栏：

- 推荐
- 最新
- 同城
- 求职内推
- 行业咨询
- 买卖交易

轻筛选：

- 城市
- 远程方式
- 行业

筛选不应占据过多首屏空间。首版可以保留现有筛选能力，但视觉上收敛成更轻的工具区。

排序：

- 推荐
- 最新

“推荐”使用推荐分排序。“最新”按发布时间倒序排序。

## 5. 卡片信息结构

每张卡片固定由 5 层组成：

1. 类型标签。
2. 主标题。
3. 类型关键信息。
4. 发布者可信信息。
5. 互动与状态。

卡片约束：

- 主标题最多 2 行，超出省略。
- 类型关键信息最多 3 个点。
- 发布者信息只展示 1 行。
- 心形图标和数字固定在卡片底部区域。
- 买卖交易有图片时展示封面图；其他类型不强制图片。
- 卡片不展示完整 `description`，只展示由类型模板挑出的关键信息。

## 6. 类型卡片模板

### 6.1 求职内推

展示：

- 类型标签：求职内推。
- 主标题。
- 目标岗位。
- 目标行业。
- 城市/远程。
- 希望获得的帮助摘要。
- 发布者：昵称、已认证、行业或城市。
- 心形图标、数量、有效期。

优先展示逻辑：

- `details.targetRole`
- `details.targetIndustry`
- `city` / `remote`
- `details.helpWanted`

### 6.2 行业咨询

展示：

- 类型标签：行业咨询。
- 主标题。
- 咨询方向。
- 具体问题摘要。
- 期望交流方式。
- 发布者：昵称、已认证、行业或城市。
- 心形图标、数量、有效期。

优先展示逻辑：

- `details.topic`
- `details.questions`
- `details.preferredFormat`

### 6.3 买卖交易

展示：

- 封面图，取第一张图片。
- 类型标签：买卖交易。
- 主标题。
- 价格或交换方式。
- 交易/发货方式。
- 城市。
- 图片数量标识。
- 发布者：昵称、已认证、城市。
- 心形图标、数量、有效期。

无图片时：

- 使用文字卡片，不显示空图片占位。
- 仍展示价格、交易方式和城市。

优先展示逻辑：

- `images[0]`
- `details.price`
- `details.deliveryMethod`
- `city`
- `images.length`

### 6.4 约稿委托

展示：

- 类型标签：约稿委托。
- 主标题。
- 委托内容。
- 预算。
- 期望交付时间。
- 发布者：昵称、已认证、行业或城市。
- 心形图标、数量、有效期。

优先展示逻辑：

- `details.commissionContent`
- `details.budget`
- `details.deadline`

### 6.5 本地互助

展示：

- 类型标签：本地互助。
- 主标题。
- 互助事项。
- 地点/区域。
- 时间窗口。
- 发布者：昵称、已认证、城市。
- 心形图标、数量、有效期。

优先展示逻辑：

- `details.helpTask`
- `details.area`
- `details.timeWindow`

### 6.6 其他

展示：

- 类型标签：其他。
- 主标题。
- 事情类型。
- 希望对方怎么帮。
- 回报方式。
- 发布者：昵称、已认证、城市或行业。
- 心形图标、数量、有效期。

优先展示逻辑：

- `details.requestKind`
- `details.helpWanted`

## 7. 心形反馈

交互：

- 卡片底部显示心形图标和数字。
- 未点状态为空心。
- 点过状态为实心。
- 点击后数字立即乐观更新。
- 再次点击取消，数字减少。
- 不出现“点赞”二字。
- 未登录用户点击时，引导登录。
- 已登录但未认证用户可以点心形，因为心形不暴露联系方式，也不触发对接。

首版权限建议：

- 已登录用户可以点心形。
- 未登录用户不能点。
- 被禁用用户不能点。
- 不要求已认证，因为心形只是轻反馈，不暴露联系方式、不触发对接。

风控：

- 同一用户对同一委托最多一个心形。
- 发布者可以给自己的委托点心形，但该行为不进入推荐加权，避免自顶。
- 被下架、过期、未审核通过的委托不能新增心形。

数据：

- `request_reactions`
- 字段：`id`, `requestId`, `userId`, `createdAt`
- 唯一约束：`requestId + userId`

API：

- `POST /api/requests/:id/reaction`
- `DELETE /api/requests/:id/reaction`
- 公共列表和详情返回：
  - `reactionCount`
  - `reactedByMe`

## 8. 推荐排序

首版推荐分为可解释排序，不做黑盒个性化。

推荐分由以下部分组成：

- 新鲜度分。
- 类型基础权重。
- 城市/行业匹配分。
- 心形数量分。
- 收藏数量分。
- 联系申请数量分。
- 发布者名片完整度分。
- 风险惩罚。

建议公式：

```text
score =
  freshnessScore
  + typeWeight
  + matchScore
  + log(1 + reactionCount) * 2
  + log(1 + favoriteCount) * 4
  + log(1 + applicationCount) * 5
  + profileCompletenessScore
  - riskPenalty
```

字段解释：

- `freshnessScore`：越新越高，随时间衰减。
- `typeWeight`：求职内推、行业咨询略微加权，但不长期压制其他类型。
- `matchScore`：用户使用频道、城市、行业筛选时产生。
- `reactionCount`：心形数，权重较轻。
- `favoriteCount`：收藏数，比心形更强。
- `applicationCount`：联系申请数，代表真实对接意图。
- `profileCompletenessScore`：发布者名片完整度。
- `riskPenalty`：举报、即将过期、重复内容等惩罚。

首版实现约束：

- SQLite 中可以先用 SQL 聚合和应用层排序完成。
- 推荐分不需要持久化。
- 过期、下架、未审核通过内容不进入公共推荐。
- 推荐排序必须稳定：分数相同时按 `createdAt DESC, id DESC`。

## 9. 频道规则

推荐：

- 默认频道。
- 使用推荐分排序。
- 包含所有公开有效委托。

最新：

- 按发布时间倒序。
- 包含所有公开有效委托。

同城：

- 优先展示与用户城市相同的委托。
- 如果用户没有城市，则提示可在我的名片完善城市。
- 同城频道内仍按推荐分或时间排序，首版建议按推荐分。

求职内推：

- 只展示 `job_referral`。
- 重点展示岗位、行业、城市/远程。

行业咨询：

- 只展示 `industry_consulting`。
- 重点展示咨询方向、具体问题、交流方式。

买卖交易：

- 只展示 `trade`。
- 有图片时卡片视觉更强。
- 继续展示交易风险提醒入口或小标识。

## 10. API 设计

公共列表：

- `GET /api/requests`
- 新增 query：
  - `channel`
  - `sort`
- `channel` 允许：
  - `recommended`
  - `latest`
  - `nearby`
  - `job_referral`
  - `industry_consulting`
  - `trade`
- `sort` 允许：
  - `recommended`
  - `latest`

响应每条委托新增：

- `reactionCount`
- `reactedByMe`
- `recommendationScore` 不返回给前端，首版只在服务端排序中使用。

详情：

- `GET /api/requests/:id`
- 返回 `reactionCount` 和 `reactedByMe`。

心形反馈：

- `POST /api/requests/:id/reaction`
- `DELETE /api/requests/:id/reaction`

返回：

```json
{
  "reactionCount": 12,
  "reactedByMe": true
}
```

## 11. 前端设计

FeedPage 拆分建议：

- `FeedPage`
- `FeedChannels`
- `FeedFilters`
- `RequestCard`
- `RequestCardFacts`
- `ReactionButton`

状态：

- 当前频道。
- 当前排序。
- 筛选项。
- 请求列表。
- loading / error / empty。
- 心形 mutation 状态。

交互：

- 切换频道后重拉列表。
- 切换排序后重拉列表。
- 筛选后重拉列表。
- 点心形时乐观更新卡片状态。
- 心形失败时回滚并显示轻提示。

移动端布局：

- 双列瀑布式卡片流。
- 卡片宽度稳定，不因心形数字变化而跳动。
- 买卖交易图片使用固定比例。
- 文本最长 2 行截断。
- 底部导航不遮挡最后一行卡片。

桌面后台不在本轮调整。

## 12. 测试范围

后端测试：

- `request_reactions` 表存在。
- 同一用户对同一委托只能有一个心形。
- 未登录不能点心形。
- 被禁用用户不能点心形。
- 不能给隐藏、过期、下架、未审核委托新增心形。
- 列表返回 `reactionCount`。
- 登录用户列表返回 `reactedByMe`。
- 推荐排序按新鲜度、心形数、收藏数、联系申请数综合排序。
- 最新排序按 `createdAt DESC, id DESC`。
- 频道筛选返回正确类型。

前端测试：

- 万事广场显示频道栏。
- 默认频道是推荐。
- 可切换最新、同城、求职内推、行业咨询、买卖交易。
- 卡片使用双列卡片流容器。
- 每类卡片展示对应关键信息。
- 心形按钮显示图标和数字，不出现“点赞”文案。
- 点击心形后变实心并更新数字。
- 失败时回滚。
- 买卖交易卡片显示封面图。
- 求职内推和行业咨询卡片突出岗位/方向信息。

端到端测试：

- 用户登录后进入万事广场。
- 切换买卖交易频道。
- 点心形后数字变化。
- 切换最新排序后列表仍可浏览。

## 13. 验收标准

- 万事广场不再是单列列表，而是移动端友好的双列卡片流。
- 用户能通过频道快速进入目标委托类型。
- 推荐/最新排序可切换。
- 心形图标可点、可取消、显示数量，不出现“点赞”二字。
- 每类委托卡片都能直观看到最关键的 2-3 个信息点。
- 买卖交易有图时能明显展示封面图。
- 求职内推、行业咨询不会被买卖交易图片流淹没。
- 过期、下架、未审核内容不参与公共推荐。
- 全量测试、build、e2e 通过。
